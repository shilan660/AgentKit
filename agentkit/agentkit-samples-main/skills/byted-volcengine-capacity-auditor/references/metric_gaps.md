# 监控缺口补齐路径

当现有脚本只能拿到资源清单，但拿不到完整监控口径时，优先按下面顺序补齐，而不是直接放弃。

## 通用原则

1. 统一优先走云监控 `GetMetricData`
2. 先确认 `Namespace / SubNamespace / MetricName / Dimensions`
3. 先在控制台确认监控项存在，再回填到脚本
4. 如果控制台有、OpenAPI 也支持，就补脚本
5. 如果控制台都没有，或者产品文档明确说明不上报，就把它标记为“当前不可自动获取”

## 通用接口

统一使用云监控接口：

- `Action=GetMetricData`
- 关键参数：
  - `Namespace`
  - `SubNamespace`
  - `MetricName`
  - `Instances`
  - `Period`

实例维度通常至少需要：

- `ResourceID`

如需更细粒度查询，再补可选维度，例如监听器、节点、磁盘等。

## 推荐排查步骤

### 1. 先在控制台确认产品监控页

确认这个产品在控制台是否已经展示了目标指标：

- 如果控制台已有该指标，记录它对应的产品、维度和时间粒度
- 如果控制台没有该指标，不要直接猜 `MetricName`

### 2. 再看云监控指标文档

优先从云监控指标文档确认：

- `Namespace`
- `SubNamespace`
- `MetricName`
- `Dimensions`

### 3. 最后补到脚本

脚本里统一通过 `safe_fetch_metric_summary(...)` 接入，避免接口报错时中断全量巡检。

## CLB

当前目标缺口主要是：

- 带宽
- 并发连接
- 新建连接
- 七层 `QPS`

补齐方式：

1. 优先在云监控控制台查看 `VCM_CLB` 指标文档
2. 先补实例维度监控，再看监听器维度是否需要额外维度字段
3. 如果用户要判断规格是否吃满，还要结合实例规格上限做使用率换算

注意：

- 并不是所有 `CLB` 计费模式都会把这几类指标上传到云监控
- 如果是按使用量计费 `CLB`，文档明确说明并发连接使用率、新建连接使用率、`QPS` 使用率、出入带宽使用率这四个指标不会上传到云监控

这类场景下应输出：

- 当前计费模式下，目标指标无法通过云监控自动获取
- 可以改为看实例侧流量、规格上限、访问日志或人工校验
- 不给结论性自动扩容判断

## RDS MySQL

当前目标缺口主要是：

- 历史 `CPU`
- 历史 `Mem`
- 历史 `Disk`
- 更细粒度的连接、吞吐、慢 SQL 和热点判断

补齐方式：

1. 先确认 `VCM_RDS_MySQL` 的云监控指标
2. 重点查下面几类指标是否可直接用于巡检：
   - `CpuUtil`
   - `MemUtil`
   - `DiskUtil` 或 `DiskUsageBytes`
   - `QPS`
   - 连接数类指标
3. 注意 `SubNamespace` 可能随产品演进发生变化，不要写死旧口径
4. 如果云监控提供的是资源监控、代理监控、引擎监控等不同分组，需要分别映射后再决定默认使用哪组

如果实例侧控制台能看到指标，但脚本拉不到，优先检查：

- `SubNamespace` 是否写错
- `MetricName` 是否已变更
- 是否需要节点维度而不是实例维度
- 是否因为时间范围和粒度不匹配导致无数据

## 什么时候不能补

以下情况不要硬补：

- 产品文档明确说明该指标不上报
- 指标仅控制台可见，但公开 `OpenAPI` 没有对应查询口径
- 当前账号没有该产品的监控权限
- 当前资源状态不满足监控采集条件，例如实例刚创建、关停、升级中

## 输出要求

一旦监控缺口仍未补齐，报告里要明确写：

1. 缺的是哪个指标
2. 为什么缺
3. 当前是否可以通过 `GetMetricData` 自动补齐
4. 如果不能自动补齐，替代证据是什么
5. 因此哪些扩缩容结论不能下
