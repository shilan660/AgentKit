# 趋势判断路由

当用户的问题已经不是“先看当前资源怎么样”，而是“未来会不会有风险、什么时候该扩容、预算会怎么走”，就进入这一段逻辑。

## 适用触发词

- 容量预测
- 未来 7 天 / 15 天 / 30 天风险
- 扩容时点
- 预算趋势
- 基于巡检结果继续判断
- 趋势分析
- 明年客户数增长 `10%`，资源应该怎么扩容
- 某条链路业务量上涨后，ECS / RDS / CLB 要不要扩容
- 如果订单量、请求量、活跃用户继续增长，应该提前扩多少

## 输入优先级

1. 优先使用当前 skill 已经产出的 JSON 巡检结果
2. 如果用户已经提供时序数据，直接消费用户数据
3. 如果用户给了业务增长假设（例如客户数、订单量、QPS、并发、带宽增长 `10%`），把它当成场景输入
4. 如果既没有巡检结果也没有时序数据，先回到主流程做巡检

## 默认方法

当用户没有指定复杂模型时，使用保守方法：

1. 看近 30 天或近 90 天趋势
2. 优先看 `avg / p95 / max`
3. 判断趋势是否持续上升
4. 用 `p95 + 安全冗余` 给建议，不把均值当上限

## 链路级场景建模规则

如果用户问的是“明年某某链路客户数增长 `10%`，资源应该怎么扩容”，不要默认按资源线性外推，按链路级场景建模处理：

1. 先明确链路边界：
   - 入口层：`CLB / API Gateway / Ingress`
   - 计算层：`ECS / 容器 / 应用节点`
   - 数据层：`RDS MySQL / Redis / MQ`
   - 依赖层：缓存、异步队列、第三方接口
2. 明确增长驱动和承压点：
   - 增长的是客户数、活跃用户数、订单量、QPS、连接数还是带宽
   - 增长首先打到哪一层，是入口连接、应用 CPU、数据库连接、磁盘还是慢 SQL
3. 先识别当前瓶颈，而不是先报扩容结论：
   - 看哪一层最接近阈值或最容易先耗尽
   - 看该层是否已经出现延迟抬升、错误、连接堆积、热点 SQL、缓存穿透等信号
4. 按链路顺序给扩容建议：
   - 如果入口先吃满，优先补 `CLB` 规格、带宽、连接和 QPS 口径
   - 如果应用层先吃满，优先补 `ECS` 规格或节点数
   - 如果数据库先吃满，优先判断是连接、CPU、存储、慢 SQL 还是读写压力
5. 只有在缺少容量模型时，才允许做保守比例估算，并明确这是场景估算：
   - 默认假设业务增长与部分资源压力近似相关
   - 不默认假设所有资源都会同步增长
   - 不把线性估算写成确定性预测

## 链路建模所需输入

优先收集下面这些信息；拿不到时，明确写入“前提假设”和“数据缺口”：

- 这条链路的入口是什么，核心请求路径是什么
- 当前瓶颈更可能在 `CLB / ECS / RDS / Redis / 下游`
- 近 30 天核心指标：`QPS / 延迟 / 错误率 / CPU / 内存 / 连接 / 带宽 / 磁盘`
- 是否存在明显峰谷、活动流量、批处理、缓存命中率变化
- 是否有实例规格上限、连接上限、QPS 上限或磁盘增长约束

## 输出要求

对于链路级问题，输出顺序固定为：

1. 链路假设
2. 当前瓶颈判断
3. 增长后最可能先到达上限的层
4. 扩容顺序
5. 验证指标
6. 前提假设
7. 数据缺口

## 业务增长类问题的推荐写法

当用户给出业务增长目标时，优先用下面这种表达：

```markdown
基于当前近 30 天巡检结果，先把该业务拆成入口层、应用层、数据库层三段来判断。

- 如果增长主要体现在新建连接、并发连接和七层请求数，先看 `CLB` 是否接近规格上限，再决定是否升规格
- 如果增长主要体现在应用 CPU、内存、线程池饱和，优先考虑 `ECS` 横向扩容或升配
- 如果增长主要体现在 `QPS / 连接数 / 慢 SQL / 存储增长`，优先判断 `RDS MySQL` 是否为主瓶颈

结论不要写成“客户数增长 `10%`，所有资源都扩 `10%`”。
要写成“哪一层先扩、为什么、扩完看哪些验证指标、哪些前提假设尚未验证”。
```

## 输出结构

```markdown
# 容量预测报告
## 摘要
## 风险判断
## 未来 7/15/30 天建议
## 预算方向建议
## 前提假设
## 数据缺口
```

## 解释规则

- 对预测结论给出 `高置信度 / 中置信度 / 低置信度`
- 如果只有 30 天数据，只给保守判断
- 如果指标缺失，明确写“无法判断”，不要硬推
- 如果资源长期低水位，优先给成本优化建议，而不是制造扩容结论
- 如果用户给的是业务增长目标，明确区分“基于监控趋势的判断”和“基于场景假设的估算”
- 如果是链路级问题，优先回答“哪个环节先成为瓶颈”，不要先回答“整体扩多少”

## 当前 V1 边界

- `ECS`：可基于近 30 天 CPU、内存、磁盘使用率做保守趋势判断
- `RDS MySQL`：当前更适合基于 `QPS / ConnUsage + 当前快照` 做方向判断
- `CLB`：只有补齐带宽、连接数、新建连接数、QPS 使用率后，才给预测结论
